#### Z35-5. ✅ Done — Ctrl+Z / Ctrl+Y перестают работать после нескольких нажатий

**Source:** «Пока тестировал, несколько раз сработали Ctrl+Z и Ctrl+Y. Потом перестали срабатывать.»

**Context:** Веб-версия (браузер). Симптом — intermittent: сначала работает, потом отказывает без видимой причины. Воспроизводится на реальных данных в процессе работы.

**Likely causes to investigate:**
- Фокус покидает компонент `EditorPage` (клик по панели, модал, контекстное меню) → обработчик `keydown` на `document` перестаёт получать события, если где-то вешается `stopPropagation` или `preventDefault`.
- Undo-стек исчерпан (маловероятно при «нескольких нажатиях», но проверить).
- При открытии и закрытии контекстного меню / модала фокус не возвращается на документ → браузер перехватывает `Ctrl+Z` для редактирования текста в поле ввода.

**Required:**
- Воспроизвести сценарий: назначить урок → Ctrl+Z (работает) → открыть контекстное меню → закрыть → Ctrl+Z (не работает?).
- Убедиться, что после закрытия любого модала / контекстного меню фокус явно возвращается на `document.body` (или глобальный `keydown`-обработчик не зависит от фокуса вовсе).
- Добавить отладочный лог (временно) для подтверждения, что `keydown` доходит до обработчика.
- Добавить регрессионный тест: после закрытия контекстного меню Ctrl+Z должен по-прежнему отменять последнее действие.

**Priority:** High

#### Z35-6. ✅ Done — Фамилии учителей искажаются при сохранении (Chrome, домашний ПК)

**Source:** «Загадочное сообщение от завуча началки. У неё на домашнем компьютере в данных коверкаются фамилии учителей. Если открыть редактирование, фамилия написана правильно. Сохраняет, и видна исковерканная. Открывает в Chrome.»

**Context:** Воспроизводится только на одном конкретном устройстве (домашний ПК, Chrome). В модале редактирования имя отображается корректно; после сохранения в списке показывается искажённый вариант.

**Likely causes to investigate:**
- Несовпадение кодировок: Chrome на Windows иногда использует другую нормализацию Unicode для кириллицы. Если в IndexedDB хранятся строки в NFC, а браузер читает в NFD (или наоборот) — при отображении буквы могут «разбиваться» на базовый символ + диакритик.
- Автозамена или автокоррекция браузера/ОС подменяет символы при вводе в `<input>` (проверить: воспроизводится ли без автокоррекции?).
- Шрифт не содержит нужных глифов → браузер делает fallback, который визуально выглядит как «коверканье», но данные в БД правильные.
- Баг в логике сохранения: поле попадает в неправильный компонент объекта или обрезается.

**Required:**
- Получить от пользователя: скриншот исковерканной фамилии + версию Chrome + ОС.
- Проверить: если открыть DevTools → Application → IndexedDB → teachers — имя там правильное или нет?
- Если в БД правильно, но на экране плохо — проблема в рендеринге/шрифте; если в БД тоже плохо — проблема в сохранении.
- Добавить нормализацию строк `str.normalize('NFC')` при сохранении в `TeacherEditModal` и в `importExcel`.

**Priority:** High

#### Z33-2. ✅ Done — Регрессия: заверение нарушения сбрасывается при изменениях в расписании

**Source:** «По-прежнему иногда загорается тревога о заверённом нарушении.»

**Required:**

### INF-SAVE — Tauri: кнопка «Сохранить» показывает диалог выбора файла

**Текущее:** `downloadJson()` использует `<a download>` — в Tauri файл молча сохраняется в папку Downloads без каких-либо вопросов. Пользователь не знает куда сохранился файл.

**Требуется:** В Tauri-сборке кнопка «Сохранить» должна открывать нативный диалог «Сохранить как» с предложенным именем файла. Пользователь выбирает папку и имя. В браузерной версии — поведение без изменений (`<a download>`).

**Реализация:**
- Добавить Rust-плагины `tauri-plugin-dialog` и `tauri-plugin-fs` в `Cargo.toml` и зарегистрировать в `lib.rs`
- Добавить разрешения в `tauri.conf.json`: `dialog:allow-save` + `fs:allow-write-text-file`
- В `import-export.ts` добавить `saveJsonFile(data, defaultFilename)`:
  - Tauri-путь: `import('@tauri-apps/plugin-dialog').save()` → если пользователь не отменил → `writeTextFile(path, data)`
  - Браузерный путь: существующий `downloadJson()` без изменений
- Оба вызова (`AppHeader.tsx:handleExportJson` и `StartPage.tsx:handleExportJson`) заменить на `saveJsonFile`

**Priority:** High (Desktop UX blocker) | **Status:** ✅ Done

### INF-WEB — Веб-сборка РШР: архив распаковывается в папку

**Текущее:** `release.yml` упаковывает `dist/index.html` командой `zip -j` — архив распаковывается в текущую директорию без подпапки.

**Требуется:** Аналогично fix'у в РН (v1.12.1): создавать подпапку `РШР-vX.Y.Z/`, помещать туда `index.html`, упаковывать как `zip -r`.

**Реализация:** В `.github/workflows/release.yml` шаг «Package web app»:
```yaml
FOLDER="РШР-${GITHUB_REF_NAME}"
mkdir -p "$FOLDER"
cp dist/index.html "$FOLDER/index.html"
zip -r "${FOLDER}.zip" "$FOLDER/"
```
